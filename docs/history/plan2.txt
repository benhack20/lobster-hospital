## 二、基于研究内容梳理的“体检项清单”（自检模块设计）

下面这些是从各类安全指南、性能优化文和故障排查文里抽出来，最适合由 skill 自动执行或辅助执行的检查项。

### 1. 环境与版本体检

**要查什么：**

- Node.js 版本：是否 ≥ 22（官方/教程强烈推荐）[12]
- OS 与架构：内存、CPU 是否满足至少 2C/4G（生产建议 4C/8G+）[11][17]
- OpenClaw 版本：
  - 是否低于已披露 CVE 修复版本（如要求 ≥ 2026.2.25 修补 ClawJacked 网关 RCE 漏洞）[1][2]
  - 是否在近期有重大安全补丁的版本段内（2026.2.x 多次紧急修复）[2][18]

**在 skill 里如何用：**

- 通过 `exec` 工具调用：
  - `node -v`
  - `openclaw --version`
- 调用 `openclaw doctor --json`，解析其中的 version / env 字段[19]。
- 规则：
  - 若 Node < 22 → 标记为 **性能+兼容性警告**；
  - 若版本 < 官方建议修补版本 → 标记为 **高危：含已公开漏洞未修补**，强烈建议升级。

---

### 2. 配置与网关暴露安全（最关键的“锁门”检查）

研究与预警反复强调：**大量事故来自默认或错误配置导致的“裸奔”**[1][2][6][20]。

**要查什么：**

1. **gateway.host 与绑定地址**
   - 若 `gateway.host` = `0.0.0.0` 且处于公网环境 → 极高风险[6][20]；
   - 若无任何认证/防护 → 任意人可接管 agent。

2. **gateway 认证与 token**
   - 是否启用 token 鉴权；
   - 是否使用默认/弱 token；
   - 是否启用控制面板“无 token 访问”。

3. **Browser / 自动化端口暴露**
   - 浏览器 WebSocket、VNC、noVNC 等端口是否对公网开放，未加保护[2][6]。

**skill 中的实现路径：**

- 调 `openclaw security audit --json`，其中有：
  - 是否存在 `gateway_public_exposed`、`insecure_debug_flags` 等标记[21]；
- 辅助读取 `~/.openclaw/openclaw.json`（注意脱敏），检查：
  - `gateway.host`
  - `gateway.auth` / `gateway.token` / `controlUi.allowInsecureAuth` 等字段[22]。
- 根据检查结果输出“医生诊断”：
  - 例如：  
    - **诊断1：网关正暴露在公网且缺乏强认证**  
      - 证据：`gateway.host=0.0.0.0`，`security audit` 标记 high risk。  
      - 处方：建议修改为 `127.0.0.1`，或仅暴露在内网；在外层配反代+WAF；执行：
        ```bash
        openclaw security audit --fix
        openclaw doctor --fix
        ```

### 3. CVE 与安全补丁情况（版本自查）

相关研究与新闻多次指出 2026 年以来 OpenClaw 出现一系列高危 CVE，比如：

- ClawJacked：本地 WebSocket 被任意网页一键接管，进而 RCE / Token 窃取[1][2]；
- 多个 WebSocket / OAuth 相关认证绕过漏洞[23][24]；
- 配置完整性问题导致安全选项在校验失败时“悄悄退回不安全默认”[25]。

**skill 可以做的：**

- 根据 `openclaw --version` 与已知“最低安全版本”对比（在 skill 中维护一个简单的版本 → 风险级别表）：
  - 若版本 < 2026.2.25：  
    - **诊断：当前版本可能受 ClawJacked 等高危漏洞影响**；
  - 若在某些已知有配置回退 bug 的版本段：
    - 提示：需要特别检查安全相关字段是否生效。

### 4. Skill / ClawHub 供应链投毒检查

各大安全公司与博客披露：  

- ClawHub 上曾被发现数百到上千个恶意 skills（ClawHavoc 供应链攻击等），部分专门窃取密钥、下载后门程序、部署挖矿[3][4][26]。
- 安全最佳实践建议：企业环境**禁止直接访问 ClawHub**，或只用自建“白名单 skill 仓库”[27]。

**skill 中可做的自查逻辑：**

- 列出当前安装的 community skills（调用 `openclaw skills list` 或扫描 `~/.openclaw/skills`）；
- 针对敏感关键词简单 heuristic：
  - 名称 / 描述中含有“wallet / crypto / shell / root / admin-only”等；
  - 来源非官方 / 无签名 / 无文档。
- 若检测到来自 ClawHub 且未知来源、近期安装的技能：
  - **诊断：存在潜在供应链风险的技能**
  - 处方建议：
    - 推荐将不明来源技能移除或禁用；
    - 引导用户参考“只从可信清单安装技能”的做法；
    - 给出简单命令示例：
      ```bash
      openclaw skills disable <skill-name>
      ```

> 高级版可以接入外部“技能黑名单清单”，但 v1 可以先做规则+人工判断提示。

### 5. 凭证与 API Key 安全自查

多篇安全指南指出：**OpenClaw 的 API Key、OAuth Token 等很容易因为配置习惯和日志而泄露**[5][6][28]。

**推荐的 skill 检查点：**

1. **明文凭证位置**：
   - `openclaw.json`、`auth-profiles.json` 中是否直接出现 `sk-`、`AKIA` 等典型 key 前缀；
   - `.env` 是否被误纳入工作空间或日志中。

2. **日志泄露风险**：
   - 检查 `~/.openclaw/logs/*.log` 中是否包含类似密钥模式（只做采样+脱敏正则检查）；
   - 注意与近期 bug 对应：老版本 `doctor` / 日志导出曾经泄露过敏感字段[19][29]。

3. **建议行为**：
   - 将 Key 迁移到环境变量或外部密钥管理（如 1Password、Vault 等）[5][28]；
   - 启用最新版中提供的“日志脱敏”、“OTEL 导出前自动 redact”等特性[18]。

**skill 输出方式**：

- 用“医生口吻”解释：
  - 你的“血型卡”（API Key）现在是写在门口白板上（配置明文）；
  - 建议收进保险柜（环境变量 / 密钥管理服务）。

### 6. 提示词注入与工具滥用自查

研究与官方 RFC 指出：OpenClaw 非常容易受提示词注入影响，尤其是当 agent 拥有**真实文件系统、命令行和网络工具**时[7][30]。

**skill 可检查的点：**

1. 工具权限：
   - `tools.fs.workspaceOnly` 是否为 `true`（推荐）[31]；
   - `exec` 工具是否在面向公网的 agent 中启用；
   - sandbox.mode 是否为 `non-main` 或 `all`，而不是关闭沙箱[31][32]。

2. 浏览器访问策略：
   - SSRF 策略是否为安全默认（例如更新后改为 `trusted-network`）[18]；
   - 是否允许任意网页执行带有“读本地文件”的指令。

3. 长期记忆 / RAG 设置：
   - 是否将未经筛查的外部内容自动写入长期记忆，再被后续指令调用——这是提示注入的“持久化形态”[30]。

**skill 里的引导方式：**

- 当检测到：
  - `tools.fs.workspaceOnly = false` 且 `sandbox.mode = main` 且 agent 面向公网；
- 输出：
  - **诊断：文件系统与执行工具权限过大，非常容易被提示注入劫持做危险操作**；
  - 给出参考配置：
    ```jsonc
    {
      "tools": {
        "fs": { "workspaceOnly": true },
        "exec": { "enabled": false }
      },
      "sandbox": { "mode": "non-main" }
    }
    ```
- 同时建议用户配合“提示注入防护 skill / 模板”：对外部网页内容在调用工具前做二次审查。

### 7. 性能与 Token 消耗自查

多篇实践文和官方/社区指南都点出：**OpenClaw 默认就很“能吃”，如果不调优，会产生巨额 token 和内存成本**[8][9][10][33]。

**skill 建议做的检查项：**

1. **上下文与 TTL 策略**
   - 是否使用默认长 TTL（如 1 小时甚至更久）导致无效对话堆积[8]；
   - 是否启用自动压缩（/compact）、分会话（/new）、重置话题（/reset 等）[34][35]。

2. **模型路由配置**
   - 是否所有任务都用最贵模型（如 GPT-4/Claude Opus）[9][33]；
   - 是否有根据任务类型拆分：重推理 / 轻问答 / 日常操作 / 本地模型等。

3. **本地模型与缓存**
   - 是否使用 Ollama / 本地 LLM 处理大量轻量任务[9][33]；
   - 是否启用 provider 的 prompt caching / 本地缓存策略[33]。

4. **监控与报表**
   - 是否使用 token 使用情况统计（如社区的 model-usage skill 等）[36]；
   - 是否设置“每月/每天 token 上限预警”。

**skill 输出的“体检报告”可以包含：**

- 抓出典型浪费点，比如：
  - 每次都带一整份项目历史（数万 token）去问一个小问题；
  - 日常 chat 也走最贵模型；
- 然后给出 3–5 条高收益建议，并链接到具体配置样例。

### 8. 运行健康与错误模式自查

从故障排查文章里可以整理出一套**错误模式 → 诊断 → 命令**的 mapping，非常适合变成 skill 的“症状识别”部分[15][16][37]。

示例：

| 症状 / 错误码 | 可能原因 | 建议检查与命令 |
| --- | --- | --- |
| WebSocket `1008`：policy violation / device identity required | gateway token 缺失或不匹配[15] | `openclaw doctor --generate-gateway-token`，重启 gateway；重新在 Dashboard/TUI 里粘贴含 token 链接 |
| WebSocket `4008`：Connect failed / Auth failed | 设备 token 过期，或 Dashboard 存了旧 token[15] | 使用 `openclaw dashboard --no-open` 生成新 token 链接；更新浏览器保存的 token |
| HTTP `401`：authentication_error: Invalid bearer token | 模型 API Key 错、被吊销或权限不足[16][38] | 检查 `openclaw models auth setup-token`；确认 Key 来源与权限 |
| HTTP `429`：rate_limit_error | 模型提供方速率限制 / 额度不足[16][39] | 降低并发、调整重试间隔；使用 fallback 模型；或增加订阅额度 |
| `MODULE_NOT_FOUND` / `@openclaw/core` 找不到 | Node 版本不合要求 / 依赖安装失败[12] | 使用 nvm 切到 Node 22+；重装依赖；检查网络或镜像 |

**skill 层实现：**

- 在自检时抓取最近日志（按时间窗口、只看 ERROR / WARN 行），抽取常见错误模式；
- 将这些模式映射到“医生诊断条目”：
  - 例如：“你最近反复出现 1008/4008 错误，这通常意味着网关 token 有问题。请按以下步骤操作……”。
